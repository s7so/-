0.1 - Problem Statement

  • ما المشكلة؟ — اكتبها بوضوح.
  • لماذا الآن؟ — سبب الإلحاح أو الفرصة.
  • من سيستفيد؟ — وصف موجز للفئة المستهدفة.

0.2 - Value Proposition

جملة واحدة تشرح القيمة للمستخدم (طفل 10 سنوات).

0.3 - Success Metrics (KPIs)

  • KPI‑1 (Usage): %DAU/MAU أو عدد المستخدمين النشطين خلال 7 أيام.
  • KPI‑2 (Performance): median API latency ≤ 200 ms.
  • KPI‑3 (Reliability): SLO مثال: 99.9% availability per 30 days.
  • KPI‑4 (Business): conversion rate أو retention.

أضف قيماً عددية واضحة لكل KPI حسب المنتج.

Roleمثال (اسم/وظيفة)RACI

  • تغييرات النطاق الكبيرة (≥ 20% effort) تحتاج Sign‑off من Sponsor.

0.5 - Prioritization (مصفوفة قرار سريعة)

RICE (Recommended)
  • Reach (R): عدد المستخدمين المتأثرين خلال فترة.
  • Impact (I): (3 = كبير، 2 = متوسط، 1 = صغير).
  • Confidence (C): % الثقة في التقدير.
  • Effort (E): شخص‑أسبوع.

RICE score = (R × I × C) / E

Betting table (Shape‑Up)
BetOwnerScope summaryAppetiteConfidenceDecision
Bet‑01POFeature X short pitch2 weeksHigh✅

استخدم RICE للعناصر الكثيرة والمجدولة؛ وBetting table للرهانات الكبيرة في الدورة.

0.6 - Security & Privacy Quick‑Check (DoR must‑have)

أجب نعم/لا وعلّم مَن يتحمّل التنفيذ إذا لا.

  • تحديد حساسية البيانات (PII)؟ — Yes/No
  • قائمة dependencies مُحدّثة ومسحوبة عبر SCA؟ — Yes/No
  • متطلبات تشفير (at‑rest / in‑transit)؟ — Yes/No
  • خطة pen‑test قبل الإصدار العام؟ — Yes/No
  • OWASP Top‑10 ضمن acceptance gate للـ Web/API؟ — Yes/No

ضع إجراءات تصحيح (mitigations) لكل عنصر "No".

0.7 - Data Retention & Privacy

  • Retention policy: مثال: سجلات المستخدمين: 2 سنة؛ سجلات الاستخدام: 90 يوم.
  • Privacy notes: هل البيانات ستُخزن خارج المنطقة؟ هل تحتاج توافق GDPR / local law؟

2.1 – Entity / Data Model (ER)

Entity: Expense
Expense { id UUID PK, user_id UUID FK -> User.id #PII?, name String, amount Decimal, date Date, category_id UUID FK, created_at Timestamp, updated_at Timestamp }

Entity: Category
Category { id UUID PK, name String UNIQUE }

علّم الحقول الحساسة بـ #PII أو #sensitive. أي جدول يحتوي user_id يخضع لسياسة وصول.

2.2 – Relationships

  • Expense → Category : Many‑to‑One.
  • User → Expense : One‑to‑Many.

2.3 – SQL Schema Example (Postgres)

CREATE TABLE users (
  id UUID PRIMARY KEY,
  email TEXT UNIQUE NOT NULL,
  name TEXT,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT now()
);

CREATE TABLE categories (
  id UUID PRIMARY KEY,
  name TEXT NOT NULL UNIQUE
);

CREATE TABLE expenses (
  id UUID PRIMARY KEY,
  user_id UUID REFERENCES users(id) ON DELETE CASCADE,
  name TEXT NOT NULL,
  amount NUMERIC(10,2) NOT NULL CHECK (amount > 0),
  date DATE NOT NULL,
  category_id UUID REFERENCES categories(id),
  created_at TIMESTAMP WITH TIME ZONE DEFAULT now()
);

CREATE INDEX idx_expenses_user_date ON expenses(user_id, date);
  • Index على (user_id, date) لقراءات سريعة.
  • فكّر في partitioning حسب date إذا كان العدد كبيراً.

2.5 – SLI / SLO

  • GET /expenses availability SLO: 99.9%.
  • POST /expenses latency SLO: p95 ≤ 300 ms.

Error budget: إذا تم تجاوز الميزانية، فعّل freeze وتأجيل ميزات غير حرجة.

2.6 – Monitoring & Alerts

  • Request rate (rps) لكل endpoint.
  • Error rate (5xx) — threshold: >1% over 5m.
  • Latency p50/p95/p99 — alert إذا p95 > 500ms لمدّة 10m.
  • DB slow queries — >200ms.
  • SLO burn rate — alert إذا burn rate > 14x لساعة.

استخدم Prometheus + Grafana + Alertmanager، أضف Sentry.

2.7 – Caching Strategy

  • Cache GET /categories بـ TTL 1h (Redis).
  • Per‑user cache للتجميعة المكلفة (30s) مع invalidate عند الكتابة.

2.8 – Backup & Restore / DR

  • Full nightly backup + WAL archiving.
  • اختبار الاستعادة شهريًا. RTO ≤ 1h، RPO ≤ 15m.

2.9 – Data Retention & Archiving

  • سجلات مالية نشطة: سنتان في DB الأساسية.
  • أرشف الأقدم إلى Parquet على S3، وإمكانية rehydrate.

2.10 – Privacy & Compliance

  • تصنيف البيانات: PII, Financial, System.
  • Right to be forgotten: سياسات حذف/إخفاء.
  • Audit log مُجزّأ بدون بيانات حساسة كاملة.

2.11 – Access Control & Security

  • RBAC وRLS (Postgres) عند الحاجة.
  • Parameterized queries / ORM.
  • Secrets management + key rotation.

2.12 – Migration Strategy

  • Migrations (Flyway / Liquibase / TypeORM).
  • Backward compatible: add nullable → backfill → enforce.
  • Feature flags + rollback plan.

2.13 – Data Quality & Observability

  • Job شهري للتحقق من العلاقات.
  • Dashboard: null rates, duplicates, orphaned FKs.
  • Alert إذا >1% سجلات جديدة تفشل التحقق.
